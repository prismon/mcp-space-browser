Project Overview

A filesystem indexing agent that crawls directories, stores metadata in SQLite, and provides tools for exploring disk utilization — similar to Baobab or WinDirStat, but exposed entirely through the Model Context Protocol (MCP).

Design Philosophy

MCP-Only — no REST or gRPC Single static binary Resource-centric operations DAG-based resource sets Plans own indexing Live filesystem monitoring Rule-based automation

Key Capabilities

  • Filesystem Crawling — Stack-based DFS traversal with incremental updates and bottom-up size aggregation
  • Resource-Set DAG — Named collections forming a directed acyclic graph with multiple parents, enabling content-type and temporal hierarchies simultaneously
  • Unified Sources — filesystem.index, filesystem.watch, query, and resource-set sources all feed into resource-sets
  • Plans & Orchestration — Plans define what to process, how to filter, and what outcomes to produce. They own the indexing lifecycle.
  • Rule Engine — Condition-outcome automation for classifying and routing files based on media type, size, time, and path patterns
  • Live Monitoring — Real-time filesystem watching with fsnotify, automatic rule execution on file changes
  • Media Classification — Thumbnail generation, metadata extraction, and content-addressed artifact caching with SHA256
  • 42+ MCP Tools — Full coverage of navigation, querying, resource management, sources, plans, and classification

Universal MCP Access

All tools are designed to work seamlessly for:

  • Human Users via MCP client applications
  • AI Models (Claude, ChatGPT, etc.) via MCP integration
  • Automated Systems implementing MCP client protocol

Schema Details

Core database tables organized by domain:

Core Data

-- Filesystem entries (files and directories)
CREATE TABLE entries (
  id INTEGER PRIMARY KEY,
  path TEXT UNIQUE NOT NULL,     -- absolute filesystem path
  parent TEXT,                    -- parent directory path
  size INTEGER,                  -- bytes (aggregated for dirs)
  kind TEXT CHECK(kind IN ('file', 'directory')),
  ctime INTEGER,                 -- creation time (unix)
  mtime INTEGER,                 -- modification time (unix)
  last_scanned INTEGER,          -- when last indexed
  dirty INTEGER DEFAULT 0        -- needs recomputation
);

-- Content-addressed metadata artifacts
CREATE TABLE metadata (
  id INTEGER PRIMARY KEY,
  source_path TEXT NOT NULL,
  metadata_type TEXT NOT NULL,   -- thumbnail, poster, timeline
  frame_index INTEGER DEFAULT 0,
  hash TEXT UNIQUE NOT NULL,     -- SHA256 content hash
  artifact_path TEXT NOT NULL,
  created_at INTEGER
);

Resource-Set DAG

-- Named collections
CREATE TABLE resource_sets (
  id INTEGER PRIMARY KEY,
  name TEXT UNIQUE NOT NULL,
  description TEXT,
  created_at INTEGER,
  updated_at INTEGER
);

-- Set membership (entries to sets)
CREATE TABLE resource_set_entries (
  set_id INTEGER NOT NULL,
  entry_path TEXT NOT NULL,
  added_at INTEGER,
  PRIMARY KEY (set_id, entry_path)
);

-- DAG edges (supports multiple parents)
CREATE TABLE resource_set_edges (
  parent_id INTEGER NOT NULL,
  child_id INTEGER NOT NULL,
  added_at INTEGER,
  PRIMARY KEY (parent_id, child_id),
  CHECK (parent_id != child_id)
);

Plans, Sources, Rules

-- Sources feed data into resource-sets
CREATE TABLE sources (
  id INTEGER PRIMARY KEY,
  name TEXT UNIQUE NOT NULL,
  type TEXT CHECK(type IN
    ('filesystem.index','filesystem.watch','query','resource-set')),
  target_set_name TEXT NOT NULL,
  update_mode TEXT CHECK(update_mode IN ('replace','append','merge')),
  config_json TEXT,
  status TEXT, enabled INTEGER DEFAULT 1,
  created_at INTEGER, updated_at INTEGER
);

-- Plans orchestrate sources and outcomes
CREATE TABLE plans (
  id INTEGER PRIMARY KEY,
  name TEXT UNIQUE NOT NULL,
  mode TEXT CHECK(mode IN ('oneshot', 'continuous')),
  status TEXT CHECK(status IN ('active', 'paused', 'disabled')),
  sources_json TEXT NOT NULL,
  conditions_json TEXT,
  outcomes_json TEXT NOT NULL,
  created_at INTEGER, updated_at INTEGER, last_run_at INTEGER
);

-- Rules: condition-outcome automation
CREATE TABLE rules (
  id INTEGER PRIMARY KEY,
  name TEXT UNIQUE NOT NULL,
  enabled INTEGER DEFAULT 1,
  priority INTEGER DEFAULT 0,
  condition_json TEXT NOT NULL,
  outcome_json TEXT NOT NULL,
  created_at INTEGER, updated_at INTEGER
);

Data Flow: Resource Query (Recursive CTE)

How resource-sum traverses the DAG to aggregate metrics

sequenceDiagram participant Client as MCP Client participant Tool as resource-sum Tool participant QB as Query Builder participant SQL as SQL Generator participant DB as SQLite Database Client->>Tool: resource-sum("all-media", "size") Tool->>QB: Build ResourceQuery QB->>SQL: Generate recursive CTE Note over SQL: WITH RECURSIVE descendants AS (
SELECT id FROM resource_sets
WHERE name = 'all-media'
UNION ALL
SELECT e.child_id
FROM resource_set_edges e
JOIN descendants d
ON e.parent_id = d.id
) SQL->>DB: Execute query DB-->>SQL: SUM(size) across all descendants SQL-->>Tool: Aggregated result Tool-->>Client: Total size of all-media tree

Space Usage Tree Calculation

The system builds an accurate space usage tree suitable for treemap or radial visualizations using a two-phase algorithm:

Phase 1: Filesystem Crawl (DFS)

  • Stack-based depth-first traversal of the filesystem
  • Files get their actual size from filesystem metadata
  • Directories are initially inserted with size=0
  • Each entry records its parent path for hierarchy navigation

Phase 2: Bottom-Up Aggregation

-- Query all directories, deepest first (by path length)
SELECT path FROM entries
WHERE kind = 'directory' AND path LIKE root || '%'
ORDER BY LENGTH(path) DESC;

-- For each directory (deepest to shallowest):
UPDATE entries SET size = (
  SELECT COALESCE(SUM(size), 0) FROM entries WHERE parent = ?
) WHERE path = ?;

Key invariant: A directory's size always equals the sum of its direct children's sizes. This bottom-up approach ensures child sizes are computed before their parents, handling any nesting depth.

MCP Resource Templates

Resources are accessible via declarative URIs using the synthesis:// scheme. These complement the imperative tool interface.

Resource-Set Resources

  • synthesis://resource-set/{name} Get resource-set metadata
  • synthesis://resource-set/{name}/entries List all entries in the set
  • synthesis://resource-set/{name}/children List child resource-sets (DAG downstream)
  • synthesis://resource-set/{name}/parents List parent resource-sets (DAG upstream)
  • synthesis://resource-set/{name}/stats Get set statistics (count, total size)
  • synthesis://resource-set/{name}/metrics/{metric} Aggregate a specific metric

Node & Job Resources

  • synthesis://nodes/{path} Get entry metadata by filesystem path
  • synthesis://jobs/{id} Get job status and progress

Plan Resources

  • synthesis://plans/{name} Get plan definition
  • synthesis://plans/{name}/executions List plan execution history

Metadata Resources

  • synthesis://metadata/{hash} Get content-addressed artifact by SHA256 hash

Dual Access Pattern

The same data is available through both imperative (tools) and declarative (resources) interfaces:

// Imperative: MCP Tool call
{"tool": "resource-set-get", "params": {"name": "photos"}}

// Declarative: MCP Resource URI
{"resource": "synthesis://resource-set/photos"}

// Both return the same resource-set data.

Tools are best for actions and mutations. Resources are best for read-only data access and subscriptions.
